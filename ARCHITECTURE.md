# Architecture Decisions

Foundational conventions every module inherits. Decided in Sprint 1; change
deliberately, not ad-hoc.

## 1. Identifiers

- **Primary key:** `uuid` (`id`), generated by the DB.
- **Display code:** every business entity also has a `code` column ‚Äî a
  human-readable, server-generated per-entity sequence matching the frontend's
  existing IDs: `P-01` (property), `U-101` (unit), `T-01` (tenant), `L-01` (lease),
  `INV-1015` (invoice), `RC-1015` (receipt/payment), `MR-07` (maintenance),
  `TECH-01` (technician), `D-30` (document), `US-1` (user).
- **Contract bridge:** during frontend integration the API serializes `code` into
  the `id` field the frontend expects. Internally, all FKs reference the `uuid`.
  This lets us keep readable codes without leaking sequential PKs and without
  breaking the existing frontend contract.

## 2. Dates & times

- Store real `date` / `timestamptz` columns ‚Äî **never** display strings.
- API returns **ISO 8601**.
- Formatting (`"01 Jul 2026"`, `"Jul 2026"`, relative `"Today"` / `"Just now"`)
  happens at the frontend edge in `lib/format.ts` ‚Äî the one deliberate FE change.

## 3. Denormalized display fields

The frontend mock stores some computed/denormalized values. In the DB we normalize;
in the response DTO/serializer we compute the display shape so the contract holds:

| Frontend field | Storage | Response |
|----------------|---------|----------|
| `MaintenanceRequest.assignee` (name string) | FK `assigneeId` ‚Üí technician | join + serialize name |
| `MaintenanceRequest.assigneeId` | real FK | passthrough |
| `Property.typeLabel` | derive from `type` | computed in serializer |
| `Invoice.items` (`[string, number][]`) | `invoice_item` child table (or JSONB) | mapped to tuples |
| `Payment` receipt / balance effects | transactional update on invoice | ‚Äî |

## 4. API conventions

- **Response envelope:** success ‚Üí `{ success: true, data, timestamp }`;
  error ‚Üí `{ success: false, error: { code, message, details? }, timestamp }`.
  The frontend HTTP client unwraps `.data`.
- **Validation:** global `ValidationPipe` (`whitelist`, `forbidNonWhitelisted`,
  `transform`) driven by `class-validator` DTOs.
- **Serialization:** `ClassSerializerInterceptor` + `@Exclude`/`@Expose` to hide
  internal fields (e.g. UUID PK, password hash) and expose `code` as `id`.
- **Errors:** global exception filter maps to the envelope; TypeORM/HTTP errors
  normalized to stable `error.code` strings.

## 5. Persistence

- **Migrations only** ‚Äî `synchronize` is **off** in every environment.
- TypeORM `DataSource` is defined once in `src/database/data-source.ts` and reused
  by both the Nest module and the TypeORM CLI (migrations).
- Seeds run through an idempotent, dependency-ordered runner (`npm run seed`).

## 6. Auth & RBAC (Sprint 2 ‚Äî implemented)

- **Passwords:** bcrypt, cost 12 (`BCRYPT_ROUNDS`). `passwordHash` is nullable ‚Äî
  invited users have none until they redeem their invite.
- **Access tokens:** short-lived JWTs (`JWT_ACCESS_TTL`, default 15 min) carrying
  `{ sub, email, tv }`. The user is re-read from the database on every request,
  so role edits and suspensions apply immediately rather than at token expiry.
- **`tokenVersion` (`tv`):** bumped on password change, password reset,
  suspension and `logout-all`. A token whose `tv` is stale is rejected, which
  closes the window where an access token outlives the event that should have
  killed it.
- **Refresh tokens:** opaque 256-bit random values, stored as SHA-256 hashes
  (not bcrypt ‚Äî they are already high-entropy, so there is nothing to
  brute-force). Rotating: redeeming one revokes it and issues a successor.
  Presenting a spent token means it was used twice, so the user's entire
  session family is revoked.
- **Invite / reset tokens:** the same hashed, single-use, expiring mechanism,
  distinguished by a `purpose` enum. Issuing a new one retires the outstanding
  token of that purpose.
- **Guards:** `JwtAuthGuard` and `PermissionsGuard` are both registered as
  `APP_GUARD`, so every route is authenticated by default. Opt out with
  `@Public()`; opt into RBAC with `@Permissions('resource.action')`.
- **Permission catalog:** `src/common/rbac/perm-catalog.ts` is the single source
  of truth (ported from the frontend's `permCatalog`, 15 resources / 45 keys).
  Role updates validate every key against it. Built-in role definitions live in
  `src/roles/built-in-roles.ts`, shared by the seeder and bootstrap registration
  so both produce an identical set ‚Äî note their codes (`role-super`, `role-pm`)
  are literals the frontend keys on, not name slugs.
- **Account creation:** `POST /auth/register` is a bootstrap path, open only
  while the system has zero users; the account it creates becomes Super Admin.
  Thereafter accounts come from admin invites.
- **Enumeration:** login and forgot-password never reveal whether an address is
  registered ‚Äî login always runs a bcrypt comparison (against a dummy hash when
  the user is missing) so timing does not leak either.
- **Mail:** `MailService` is a logging stub; swapping in a transport means
  reimplementing that class alone.
- Audit: an interceptor writes an `AuditEntry` on mutating requests *(S7)*.
