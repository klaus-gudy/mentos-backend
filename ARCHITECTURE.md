# Architecture Decisions

Foundational conventions every module inherits. Decided in Sprint 1; change
deliberately, not ad-hoc.

## 1. Identifiers

- **Primary key:** `uuid` (`id`), generated by the DB.
- **Display code:** every business entity also has a `code` column — a
  human-readable, server-generated per-entity sequence matching the frontend's
  existing IDs: `P-01` (property), `U-101` (unit), `T-01` (tenant), `L-01` (lease),
  `INV-1015` (invoice), `RC-1015` (receipt/payment), `MR-07` (maintenance),
  `TECH-01` (technician), `D-30` (document), `US-1` (user).
- **Contract bridge:** during frontend integration the API serializes `code` into
  the `id` field the frontend expects. Internally, all FKs reference the `uuid`.
  This lets us keep readable codes without leaking sequential PKs and without
  breaking the existing frontend contract.

## 2. Dates & times

- Store real `date` / `timestamptz` columns — **never** display strings.
- API returns **ISO 8601**.
- Formatting (`"01 Jul 2026"`, `"Jul 2026"`, relative `"Today"` / `"Just now"`)
  happens at the frontend edge in `lib/format.ts` — the one deliberate FE change.

## 3. Denormalized display fields

The frontend mock stores some computed/denormalized values. In the DB we normalize;
in the response DTO/serializer we compute the display shape so the contract holds:

| Frontend field | Storage | Response |
|----------------|---------|----------|
| `MaintenanceRequest.assignee` (name string) | FK `assigneeId` → technician | join + serialize name |
| `MaintenanceRequest.assigneeId` | real FK | passthrough |
| `Property.typeLabel` | derive from `type` | computed in serializer |
| `Invoice.items` (`[string, number][]`) | `invoice_item` child table (or JSONB) | mapped to tuples |
| `Payment` receipt / balance effects | transactional update on invoice | — |

## 4. API conventions

- **Response envelope:** success → `{ success: true, data, timestamp }`;
  error → `{ success: false, error: { code, message, details? }, timestamp }`.
  The frontend HTTP client unwraps `.data`.
- **Validation:** global `ValidationPipe` (`whitelist`, `forbidNonWhitelisted`,
  `transform`) driven by `class-validator` DTOs.
- **Serialization:** `ClassSerializerInterceptor` + `@Exclude`/`@Expose` to hide
  internal fields (e.g. UUID PK, password hash) and expose `code` as `id`.
- **Errors:** global exception filter maps to the envelope; TypeORM/HTTP errors
  normalized to stable `error.code` strings.

## 5. Persistence

- **Migrations only** — `synchronize` is **off** in every environment.
- TypeORM `DataSource` is defined once in `src/database/data-source.ts` and reused
  by both the Nest module and the TypeORM CLI (migrations).
- Seeds run through an idempotent, dependency-ordered runner (`npm run seed`).

## 6. Auth (Sprint 2, recorded here for context)

- Self-hosted: bcrypt password hashes, JWT access + refresh tokens.
- RBAC from the frontend's `permCatalog`: permission key = `resource.action`
  (e.g. `lease.create`, `invoice.read`). A `@Permissions()` decorator + guard
  enforce them; roles carry a `perms: string[]`.
- Audit: an interceptor writes an `AuditEntry` on mutating requests.
